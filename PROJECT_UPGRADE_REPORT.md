# PROJECT_UPGRADE_REPORT.md — تقرير إعادة بناء المشروع الشامل (v1 → v2.0 CMS)

## 1. ملخص تنفيذي

تم تحويل المشروع من موقع تعريفي بلوحة تحكم بدائية (إدخال يدوي محدود، بيانات Hardcoded في الكود، بدون نظام صلاحيات حقيقي) إلى **منصة CMS متكاملة** تُدار بالكامل من لوحة تحكم احترافية، حيث كل بكسل تقريباً في الموقع العام (الشعار، الألوان، الخلفيات، السلايدر، القوائم، المحتوى، الطلبات) مصدره قاعدة بيانات Supabase ولا يحتاج لمس الكود البرمجي للتعديل.

## 2. التحليل الأولي — المشاكل المكتشفة في النسخة السابقة

| # | المشكلة | الأثر |
|---|---|---|
| 1 | بيانات الشركة (الاسم، التواصل، السوشيال ميديا) Hardcoded داخل `Header.tsx` / `Footer.tsx` / `layout.tsx` | أي تعديل يتطلب تعديل كود ونشر جديد |
| 2 | لا يوجد نظام صلاحيات — كل مستخدم مسجّل دخول لديه وصول كامل | خطر أمني، لا تتبع لمن قام بأي تعديل |
| 3 | لا يوجد Activity/Audit Log | استحالة معرفة من حذف/عدّل ماذا ومتى |
| 4 | لا يوجد Error Boundary على مستوى /admin أو الموقع العام | أي استثناء في صفحة واحدة يُسقط كامل الجلسة دون رسالة واضحة |
| 5 | الصور تُرفع كروابط خارجية فقط (لا يوجد API رفع حقيقي لـ Supabase Storage) | اعتماد على استضافة صور خارجية غير موثوقة |
| 6 | لا يوجد Site Builder / Theme Builder / Background Manager / Slider Builder / Page Builder / Media Library | كل تخصيص بصري يتطلب تدخل مبرمج |
| 7 | المنتجات بدون تصنيفات أو علامات تجارية أو سعر خصم أو مخزون كمي | نظام متجر مبدئي جداً |
| 8 | المشاريع بدون صور قبل/بعد، فيديو، أو حالة تنفيذ | Portfolio غير احترافي |
| 9 | لا توجد صفحة تسجل أخطاء العميل (Client-side errors) | أخطاء الواجهة تمر دون أي تتبع |
| 10 | الـ revalidation غير مضمونة لكل تعديل (بعض الصفحات تبقى بكاش قديم) | "التعديل لا ينعكس على الموقع" — وهو العرض الذي طلب المستخدم حله تحديداً |

## 3. خطة الإصلاح المتبعة

1. **إعادة بناء قاعدة البيانات بالكامل** بهيكلة قابلة للتوسع (24 جدولاً جديداً/محدثاً) مع RLS صارمة على كل جدول.
2. **فصل واضح بين القراءة العامة والكتابة الإدارية**: كل Server Action للكتابة يستخدم `createAdminClient()` (service_role، خادم فقط) بينما القراءة العامة تستخدم `createClient()` (anon، محكومة بـ RLS).
3. **استدعاء `revalidatePath()` في نهاية كل Server Action** يمسّ بيانات تظهر في الموقع العام، لضمان انعكاس التعديل فوراً دون انتظار دورة بناء جديدة.
4. **توحيد رفع الملفات** عبر `app/api/upload/route.ts` إلى Supabase Storage Bucket واحد (`media`)، مع حفظ المسار فقط في جدول `media` (لا تُخزَّن الملفات الثنائية في أي جدول).
5. **نظام صلاحيات حقيقي بخمس درجات** (Super Admin / Admin / Editor / Content Manager / Viewer) مطبّق عبر RLS (`is_admin()`, `has_role()`) وعبر تحقق إضافي صريح داخل Server Actions الحساسة (مثل إدارة المستخدمين، التي تتحقق أن المنفّذ `super_admin` قبل أي عملية).
6. **Error Boundaries على مستويين**: `app/admin/error.tsx` للوحة التحكم، `app/error.tsx` للموقع العام، و`app/global-error.tsx` كحارس أخير على مستوى التطبيق بالكامل، بالإضافة إلى `app/not-found.tsx`. كل خطأ من جهة العميل يُرسَل تلقائياً إلى `/api/log-error` ليُسجَّل في `activity_logs` بخطورة `error`.
7. **سجل نشاطات شامل (`activity_logs`)** يُكتب إليه من كل Server Action رئيسي (إضافة/تعديل/حذف خدمة، منتج، مشروع، مقال، تغيير صلاحيات، تسجيل دخول/خروج...)، مع عرضه في `/admin/logs` بفلترة حسب درجة الخطورة.
8. **نظام إشعارات داخلي (`notifications`)** يُولَّد تلقائياً عند وصول طلب/رسالة تواصل/طلب عرض سعر جديد، ويظهر فوراً عبر جرس الإشعارات في `TopBar` وصفحة `/admin/notifications`، بالإضافة إلى الإشعار الصوتي اللحظي (Supabase Realtime) للطلبات تحديداً.

## 5. التحسينات الأمنية

- Row Level Security مفعّلة على **كل** الجداول الـ24 دون استثناء.
- مفتاح `service_role` لا يصل أبداً إلى المتصفح؛ يُستخدم فقط داخل Server Actions/Route Handlers.
- API الرفع (`/api/upload`) يتحقق من الجلسة (`auth.getUser()`) قبل قبول أي ملف، ويرفض الأنواع/الأحجام غير المسموحة (حد أقصى 10MB، قائمة بيضاء لأنواع الملفات).
- `middleware.ts` يحمي كل مسار `/admin/*` و`/api/*` بفحص الجلسة قبل الوصول.
- حذف مستخدم محصور بـ Super Admin فقط، ولا يمكن لأي مستخدم حذف حسابه الخاص (حماية من القفل الذاتي خارج النظام).
- رؤوس أمان إضافية في `next.config.js` (X-Frame-Options, X-Content-Type-Options, Referrer-Policy).
- جميع نماذج الإدخال العامة (تواصل/عروض سعر/طلبات) تُعالَج عبر Server Actions مع تحقق من الحقول قبل أي إدخال لقاعدة البيانات.

## 6. تحسينات الأداء

- `force-dynamic` مضبوط على الصفحات التي تعتمد بيانات لحظية (Dashboard، صفحات الإدارة) لضمان عدم عرض بيانات قديمة، مع الاعتماد على `revalidatePath` المستهدف لباقي صفحات الموقع العام بدل إعادة البناء الكامل.
- `next/image` مفعّل لكل الصور القادمة من Supabase Storage (`remotePatterns` في `next.config.js`) لتوليد صيغ AVIF/WebP تلقائياً.
- مكوّنات تفاعلية (`'use client'`) محصورة في أصغر نطاق ممكن (أزرار، نماذج، سلايدر) بينما تبقى صفحات القراءة Server Components لتقليل الـ JS المرسل للعميل.
- `DataTable` موحّد بترقيم صفحات وبحث من جهة العميل لتفادي إعادة طلب الخادم عند كل بحث بسيط.

## 7. خطوات النشر (ملخص — التفاصيل الكاملة في DATABASE_SETUP.md)

3. إنشاء أول مستخدم Super Admin.
4. ضبط متغيرات البيئة في الاستضافة.
